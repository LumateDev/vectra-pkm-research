# 3 Экспериментальная реализация прототипа интеллектуальной системы управления персональными знаниями

Данная глава посвящена описанию экспериментальной реализации прототипа интеллектуальной системы управления персональными знаниями. В отличие от промышленных PKM-платформ, рассмотренных во второй главе, разработанный прототип не ставит целью заменить полноценные системы хранения заметок, совместной работы или управления проектами. Его назначение состоит в проверке отдельных интеллектуальных механизмов, которые могут быть использованы в персональных базах знаний: семантического поиска, поиска по ключевым словам, автоматического тегирования, автоматического определения категории и поиска похожих заметок.

Прототип реализован как небольшое исследовательское приложение на языке Python с использованием библиотеки Streamlit для пользовательского интерфейса. Основные вычислительные функции вынесены из интерфейсного слоя в отдельные модули, что позволяет использовать их не только в интерактивном режиме, но и в автоматических экспериментах. Такой подход соответствует исследовательскому характеру работы: система должна быть достаточно простой для анализа и воспроизведения, но при этом демонстрировать принципиальные возможности методов искусственного интеллекта при работе с персональными знаниями.

## 3.1 Постановка задачи экспериментального исследования

Цель экспериментальной реализации заключается в создании программного прототипа, позволяющего проверить, насколько методы семантического представления текста могут быть полезны для поиска и первичной организации персональных заметок. В рамках данной главы рассматривается не разработка промышленного программного продукта, а построение ограниченной исследовательской среды, в которой можно сравнить несколько способов обработки текстовых записей и оценить их применимость к задачам PKM.

Основной исследовательский акцент сделан на сравнении двух подходов к поиску информации. Первый подход основан на поиске по ключевым словам и использует статистическую модель TF-IDF. Данный метод выбран в качестве базового варианта, поскольку он широко применяется в задачах информационного поиска, обладает понятной интерпретацией и хорошо работает в случаях, когда запрос пользователя содержит те же термины, что и документ. Второй подход основан на семантическом поиске с использованием эмбеддингов предложений. Он позволяет представить текст заметки и поисковый запрос в виде числовых векторов и оценивать их близость не только по совпадению слов, но и по смысловому сходству.

В качестве рабочей гипотезы принято предположение о том, что семантический поиск может быть более удобным для персональных заметок, чем классический keyword search, особенно в ситуациях, когда пользователь помнит общий смысл записи, но не помнит точную формулировку. Такая ситуация типична для систем управления персональными знаниями: пользователь часто возвращается к ранее созданным фрагментам информации спустя значительное время и формулирует запрос иначе, чем исходный текст заметки.

Для проверки данной гипотезы в прототипе реализованы следующие функции:

- просмотр заметок из тестового набора данных;
- поиск по ключевым словам на основе TF-IDF;
- семантический поиск на основе sentence-transformers embeddings;
- режим сравнения keyword search и semantic search в одном интерфейсе;
- автоматическое назначение тегов на основе близости эмбеддингов;
- автоматическое определение категории заметки;
- поиск похожих заметок с использованием косинусной близости;
- автоматический экспериментальный прогон с расчётом метрик Precision@K, Recall@K, nDCG@K и MRR.

Таким образом, задача экспериментального исследования состоит не только в создании интерфейса для демонстрации работы системы, но и в построении воспроизводимого набора программных компонентов, позволяющих сравнивать разные методы поиска на одном и том же наборе данных. При этом оценка результатов имеет демонстрационный характер, поскольку используемые наборы заметок являются синтетическими и не охватывают всего разнообразия реальных персональных баз знаний.

## 3.2 Формирование тестового набора данных

Для проведения экспериментов в проекте сформированы синтетические наборы данных, имитирующие короткие PKM-style заметки. Основным набором для пользовательского интерфейса является русскоязычный файл `datasets/pkm_notes_ru.csv`, содержащий 240 заметок. Также в проекте имеются англоязычный набор `datasets/pkm_notes_large.csv` и небольшой стартовый набор `datasets/notes.csv`. В рамках данной главы основное внимание уделяется русскоязычному набору, поскольку он используется в интерфейсе прототипа по умолчанию.

Структура набора данных сохраняет поля, необходимые для последующего анализа и воспроизводимости экспериментов: `id`, `title`, `content`, `topic`, `source`. Поле `id` используется для идентификации заметки, `title` содержит заголовок, `content` содержит основной текст заметки, `topic` задаёт тематическую принадлежность, а `source` фиксирует происхождение записи. При загрузке данных в приложении дополнительно формируется служебное поле `text`, объединяющее заголовок и содержание заметки. Именно это объединённое текстовое представление используется для построения поисковых индексов и вычисления эмбеддингов.

Русскоязычный набор данных сформирован генератором `experiments/generate_russian_dataset.py`. В нём определены тематические группы, каждая из которых включает набор типовых заголовков, терминов и текстовых шаблонов. На основе этих шаблонов генерируются короткие записи, похожие на рабочие фрагменты личной базы знаний, а не на завершённые статьи. Всего в наборе выделено восемь тематических направлений по 30 заметок в каждом:

| Тема | Количество заметок | Смысловое содержание |
| --- | ---: | --- |
| `semantic_search` | 30 | поиск по смыслу, эмбеддинги, косинусная близость |
| `keyword_search` | 30 | TF-IDF, baseline, лексический поиск |
| `tagging` | 30 | автоматические теги, метки, категоризация |
| `databases` | 30 | хранение заметок, схемы данных, метаданные |
| `research_methods` | 30 | метрики, разметка релевантности, методология |
| `productivity` | 30 | личные процессы работы со знаниями |
| `software_architecture` | 30 | простая структура модулей и границы компонентов |
| `summarization` | 30 | краткие описания и предварительный просмотр заметок |

Несмотря на наличие темы `summarization` в тестовом наборе, в самом интерфейсном прототипе не реализован механизм автоматической генерации резюме. Данные этой темы используются только как часть синтетического корпуса для проверки поисковых механизмов. Это важно учитывать при описании функциональности системы: прототип осуществляет поиск, тегирование, категоризацию и подбор похожих заметок, но не выполняет генерацию новых текстов.

Для автоматической оценки поиска подготовлен файл `experiments/search_scenarios_ru.csv`, содержащий 15 поисковых сценариев. Каждый сценарий включает текст запроса, список релевантных тем и краткое описание информационной потребности. Примером такого сценария является запрос «как найти похожие идеи если я не помню точные слова», который должен приводить к заметкам темы `semantic_search`. Такой формат позволяет оценивать выдачу не на уровне ручного просмотра отдельных результатов, а на основе заранее заданной связи между запросами и тематическими группами.

Поскольку набор данных является синтетическим, его нельзя рассматривать как полную модель реального пользовательского архива. Однако для целей курсовой работы он обеспечивает несколько важных свойств: контролируемую структуру, воспроизводимость, наличие тематической разметки и возможность сравнивать разные алгоритмы поиска на одинаковых условиях. Это делает его подходящим для экспериментальной демонстрации возможностей интеллектуальной PKM-системы.

## 3.3 Архитектура экспериментального прототипа

Архитектура прототипа построена по модульному принципу, но без усложнения, характерного для промышленных веб-систем. В проекте отсутствует отдельный backend-сервис, база данных, система авторизации, многопользовательский режим и механизмы развёртывания в контейнерах. Все эти элементы не входят в задачи текущего экспериментального исследования. Вместо этого прототип организован как локальное Streamlit-приложение, которое загружает CSV-набор заметок, строит поисковые индексы и отображает результаты в пользовательском интерфейсе.

Структура проекта отражает разделение основных функций:

- `app/data` — загрузка набора заметок;
- `app/search` — реализация keyword search и semantic search;
- `app/tagging` — автоматическое тегирование и категоризация;
- `app/ui` — компоненты пользовательского интерфейса;
- `app/utils` — конфигурация, включая выбор embedding-модели;
- `experiments` — генерация данных, сценарии оценки, метрики и отчёты;
- `datasets` — синтетические наборы заметок.

Центральной точкой запуска является файл `app/main.py`. В нём описана логика Streamlit-приложения: загрузка заметок, инициализация embedding-модели, создание объектов поиска, подготовка автоматических тегов и категорий, а также построение интерфейса. Для повышения удобства работы используются механизмы кэширования Streamlit. Модель эмбеддингов кэшируется как ресурс, набор заметок кэшируется как данные, а поисковые индексы и объект тегирования также создаются один раз и переиспользуются при последующих действиях пользователя.

Загрузка данных реализована в модуле `app/data/loader.py`. Функция загрузки читает CSV-файл и проверяет наличие обязательных столбцов `id`, `title`, `content`. После этого формируется поле `text`, объединяющее заголовок и содержание. Такой подход позволяет единообразно использовать текст заметки в разных компонентах: при построении TF-IDF-матрицы, при вычислении эмбеддингов, при поиске похожих заметок и при автоматической классификации.

Отдельное значение имеет то, что поисковая логика не встроена непосредственно в Streamlit-интерфейс. Классы `TfidfKeywordSearch` и `SemanticSearch` могут использоваться независимо от визуального слоя. Именно поэтому они применяются как в интерактивном приложении, так и в скрипте автоматической оценки `experiments/evaluate_search.py`. Это делает прототип удобным для исследовательского использования: интерфейс служит для визуальной демонстрации, а экспериментальные скрипты — для получения количественных результатов.

В качестве модели эмбеддингов по умолчанию используется `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`. Модель выбирается в модуле `app/utils/config.py`, при этом предусмотрена возможность изменить её через переменную окружения `VECTRA_EMBEDDING_MODEL`. Использование multilingual-модели связано с тем, что основной демонстрационный набор данных и пользовательские запросы являются русскоязычными.

Таким образом, архитектура экспериментального прототипа представляет собой компактную исследовательскую систему, в которой данные, поиск, классификация и интерфейс разделены на простые модули. Такое решение позволяет сохранить читаемость кода и одновременно обеспечить повторяемость экспериментов.

## 3.4 Реализация механизма семантического поиска

Механизм семантического поиска реализован в модуле `app/search/semantic.py`. Его основная идея состоит в представлении каждой заметки и каждого пользовательского запроса в виде векторного embedding-представления. После этого релевантность заметки определяется как косинусная близость между вектором запроса и вектором текста заметки.

При создании объекта `SemanticSearch` набор заметок сначала приводится к последовательной индексации, после чего embedding-модель кодирует список текстов из поля `text`. Для вычисления эмбеддингов используется метод `encode` библиотеки sentence-transformers. Вызов выполняется с параметром `normalize_embeddings=True`, что нормализует векторы и делает дальнейшее вычисление косинусной близости более удобным. Полученная матрица эмбеддингов хранится в объекте поиска и используется повторно, поэтому при каждом новом запросе не требуется заново кодировать весь корпус заметок.

Алгоритм обработки пользовательского запроса включает следующие шаги:

1. Проверяется, что запрос не является пустым.
2. Запрос преобразуется в embedding с помощью той же модели, что использовалась для заметок.
3. Вычисляется косинусная близость между embedding запроса и embedding каждой заметки.
4. Индексы заметок сортируются по убыванию значения близости.
5. Возвращаются первые `top_k` результатов с идентификатором заметки, заголовком, числовой оценкой релевантности и фрагментом содержания.

Такой подход позволяет находить заметки, которые близки к запросу по смыслу, даже если в них отсутствует буквальное совпадение слов. Например, запрос о поиске «похожих идей без точных слов» может приводить к заметкам о семантическом поиске, эмбеддингах и косинусной близости, поскольку эти тексты находятся рядом с запросом в векторном пространстве.

В том же модуле реализована функция поиска похожих заметок `related_notes`. В отличие от обычного поиска, входом здесь является не произвольный пользовательский запрос, а идентификатор выбранной заметки. Система находит embedding этой заметки, сравнивает его с embedding всех остальных записей и возвращает наиболее близкие тексты, исключая саму исходную заметку. В интерфейсе данная функция используется в блоке «Похожие заметки» и показывает три semantic-соседа выбранной записи.

Таким образом, семантический поиск в прототипе используется в двух режимах. Первый режим предназначен для обработки текстового запроса пользователя. Второй режим предназначен для выявления связей между уже существующими заметками. Оба режима основаны на одном принципе: измерении косинусной близости между нормализованными embedding-представлениями текстов.

## 3.5 Реализация автоматической категоризации и тегирования

Автоматическое тегирование и категоризация реализованы в модуле `app/tagging/classifier.py`. Данный компонент также основан на embedding-представлениях и косинусной близости, однако применяется не к поисковому запросу, а к задаче первичной организации заметок.

В прототипе задан фиксированный словарь тегов: `python`, `machine learning`, `databases`, `frontend`, `backend`, `ai`, `productivity`, `architecture`, `research`, `embeddings`. Для отображения в интерфейсе часть тегов переводится на русский язык в модуле `app/ui/components.py`, например `machine learning` отображается как «машинное обучение», `ai` — как «ИИ», `productivity` — как «продуктивность». Такой подход делает внутреннюю модель достаточно простой, но позволяет демонстрировать пользователю понятные метки.

При инициализации объекта `EmbeddingTagger` система заранее вычисляет эмбеддинги для всех тегов. Затем при классификации конкретной заметки вычисляется embedding её текста, после чего рассчитывается косинусная близость между embedding заметки и embedding каждого тега. Теги сортируются по убыванию близости, и в качестве результата выбираются три наиболее близких элемента. В интерфейсе они отображаются как автоматические теги заметки.

Автоматическая категория определяется похожим способом, но вместо коротких тегов используются текстовые описания категорий. В проекте заданы следующие категории:

- `Programming`;
- `AI`;
- `Databases`;
- `Research`;
- `Productivity`;
- `Personal`;
- `Projects`.

Для каждой категории задано краткое описание на английском языке, включающее характерные слова и смысловые признаки. Например, категория `AI` описывается через artificial intelligence, machine learning, embeddings, semantic search и models, а категория `Research` — через papers, experiments, metrics, hypotheses, evaluation и methodology. Система вычисляет эмбеддинги этих описаний и сравнивает их с embedding текста заметки. Категорией считается та, для которой значение косинусной близости максимально.

Результатом работы классификатора является структура `TaggingResult`, содержащая список тегов, выбранную категорию и числовую оценку категории. В пользовательском интерфейсе отображаются теги и категория, тогда как числовая оценка категории используется внутри результата классификации и не выводится отдельно. Классификация всех заметок выполняется в `app/main.py` при подготовке данных для интерфейса. Для каждой записи вызывается метод `classify`, после чего в таблицу заметок добавляются поля `tags` и `category`.

Следует подчеркнуть, что реализованный механизм не является обученной supervised-моделью классификации. Он не использует обучающую выборку с размеченными категориями и не выполняет дообучение модели. Категоризация и тегирование осуществляются на основе близости между текстом заметки и заранее заданными текстовыми представлениями тегов и категорий. Такой метод хорошо соответствует экспериментальному характеру прототипа: он прост, воспроизводим и демонстрирует практическое применение эмбеддингов для автоматической организации персональных знаний.

## 3.6 Описание пользовательского интерфейса

Пользовательский интерфейс прототипа реализован с помощью Streamlit и ориентирован на исследовательскую демонстрацию, а не на создание сложной PKM-платформы. После запуска приложения пользователь видит заголовок «Vectra Research» и краткое описание назначения системы: сравнение keyword search и semantic search. В верхней части также размещён раскрываемый блок с инструкцией, в котором перечислены основные действия: выбор заметки, ввод поискового запроса, выбор режима поиска и просмотр похожих заметок.

Интерфейс разделён на боковую панель и две основные колонки. В боковой панели расположен список заметок из датасета. Пользователь выбирает запись по заголовку, а система использует её идентификатор для отображения содержания и поиска похожих материалов. Ниже выводится метрика размера датасета, показывающая количество загруженных заметок. Это подчёркивает экспериментальный характер приложения: пользователь работает не с произвольной личной базой данных, а с заранее подготовленным набором синтетических PKM-записей.

Левая колонка предназначена для просмотра выбранной заметки. Компонент карточки отображает номер заметки, автоматически определённую категорию, заголовок, полный текст и список автоматических тегов. Например, на экране просмотра может быть показана заметка категории «Искусственный интеллект» с тегами «эмбеддинги», «базы данных» и «машинное обучение». Ниже располагается блок «Похожие заметки», в котором отображаются semantic-соседи выбранной записи. Для каждого результата указывается место в выдаче, значение релевантности, номер заметки, заголовок и фрагмент текста.

Рисунок 3.1 — Главный экран экспериментального прототипа.

Правая колонка содержит блок поиска. Пользователь вводит текстовый запрос в поле ввода и выбирает один из трёх режимов: «Семантический поиск», «Поиск по ключевым словам» или «Сравнение». Дополнительно предусмотрен ползунок `Top-K`, который определяет, сколько результатов следует показать. В текущей реализации пользователь может выбрать от 3 до 10 результатов, значение по умолчанию равно 5.

В режиме «Семантический поиск» запрос обрабатывается через embedding-модель. Система возвращает заметки, отсортированные по косинусной близости к запросу. В интерфейсе каждый результат отображается в отдельной карточке с указанием ранга, релевантности и идентификатора заметки. Такой режим демонстрирует способность системы находить материалы не только по совпадению слов, но и по смысловой близости.

Рисунок 3.2 — Демонстрация режима семантического поиска.

В режиме «Поиск по ключевым словам» используется TF-IDF-представление текста. Пользовательская строка преобразуется тем же векторизатором, который был построен на корпусе заметок, после чего результаты сортируются по косинусной близости TF-IDF-векторов. Этот режим позволяет увидеть, какие заметки находятся при классическом лексическом поиске.

Режим «Сравнение» является особенно важным для исследовательской части работы. В этом режиме интерфейс делится на две колонки: в одной отображается выдача keyword search, в другой — выдача semantic search. Пользователь может ввести один и тот же запрос и визуально сравнить, какие результаты возвращают разные методы. Это позволяет показать различие между лексическим совпадением и смысловой близостью на конкретных примерах из набора заметок.

Компоненты отображения заметки и результатов вынесены в модуль `app/ui/components.py`. Это позволяет избежать дублирования интерфейсного кода: одна и та же функция используется для отображения результатов keyword search, semantic search и похожих заметок. При отсутствии запроса или результатов система выводит информационное сообщение с предложением ввести запрос. Такая реализация делает интерфейс минимальным, но достаточным для демонстрации основных возможностей прототипа.

## 3.7 Анализ результатов экспериментальной реализации

Для количественной оценки поисковых механизмов в проекте реализован скрипт `experiments/evaluate_search.py`. Он выполняет автоматическое сравнение keyword search и semantic search на заранее подготовленных сценариях. В качестве входных данных используются набор заметок и CSV-файл с поисковыми запросами. Для каждого сценария задаются релевантные темы, а релевантными заметками считаются все записи, чей столбец `topic` входит в список релевантных тем.

Оценка проводится для значений K, равных 3, 5 и 10. Для каждого метода и каждого запроса вычисляются следующие метрики:

- Precision@K — доля релевантных заметок среди первых K результатов;
- Recall@K — доля найденных релевантных заметок среди всех известных релевантных записей;
- nDCG@K — метрика, учитывающая не только наличие релевантных результатов, но и их положение в выдаче;
- MRR — величина, отражающая, насколько рано в выдаче появляется первый релевантный результат.

Результаты сохраняются в каталог `experiments/reports/`. Для русскоязычного набора формируются файлы `search_metrics_ru.csv`, `search_top_results_ru.csv` и `search_evaluation_summary_ru.md`. Аналогичные файлы формируются для англоязычного набора данных.

Средние результаты автоматического прогона на русскоязычном наборе данных представлены в таблице 3.1.

Таблица 3.1 — Средние метрики поиска на русскоязычном синтетическом наборе данных

| Метод | K | Precision | Recall | nDCG | MRR |
| --- | ---: | ---: | ---: | ---: | ---: |
| keyword | 3 | 0,578 | 0,058 | 0,580 | 0,656 |
| semantic | 3 | 0,689 | 0,069 | 0,659 | 0,674 |
| keyword | 5 | 0,640 | 0,107 | 0,624 | 0,656 |
| semantic | 5 | 0,640 | 0,107 | 0,634 | 0,674 |
| keyword | 10 | 0,567 | 0,189 | 0,578 | 0,656 |
| semantic | 10 | 0,593 | 0,198 | 0,602 | 0,674 |

Из таблицы видно, что на русскоязычном наборе семантический поиск показывает более высокое среднее значение nDCG для всех трёх проверенных значений K. При K=5 значение nDCG для semantic search составляет 0,634, тогда как для keyword search — 0,624. Разница не является радикальной, однако она подтверждает, что embedding-подход способен улучшать ранжирование результатов в условиях переформулированных запросов. При K=3 семантический поиск также показывает более высокую Precision@K: 0,689 против 0,578 у keyword search.

Отдельно следует отметить, что при K=5 значения Precision и Recall у двух методов совпадают, но nDCG остаётся выше у semantic search. Это означает, что оба метода в среднем находят сопоставимое количество релевантных заметок, однако семантический поиск располагает их несколько выше в выдаче. Для пользовательского поиска в PKM-системе такая особенность имеет практическое значение, поскольку пользователь чаще просматривает первые результаты и может не доходить до нижних позиций списка.

На англоязычном наборе данных преимущество семантического поиска выражено сильнее. При K=5 nDCG semantic search составляет 0,862, тогда как keyword search показывает 0,726. Precision@5 также выше у семантического подхода: 0,840 против 0,733. Эти результаты согласуются с общей гипотезой о пользе embedding-поиска, но должны рассматриваться как дополнительная демонстрация, поскольку основной пользовательский интерфейс ориентирован на русскоязычный набор.

Вместе с тем эксперимент имеет ряд ограничений. Во-первых, наборы данных являются синтетическими, поэтому они не полностью отражают неоднородность, непоследовательность и индивидуальный стиль реальных пользовательских заметок. Во-вторых, релевантность определяется по тематическим группам, а не через независимую экспертную оценку каждой пары «запрос — заметка». В-третьих, автоматическое тегирование и категоризация в текущей реализации не оцениваются отдельным набором метрик, а демонстрируются в интерфейсе как embedding-based механизм первичной организации.

Несмотря на эти ограничения, полученные результаты позволяют сделать вывод о работоспособности экспериментального прототипа. Система корректно загружает набор заметок, строит TF-IDF и embedding-представления, выполняет поиск в трёх пользовательских режимах, автоматически назначает теги и категории, а также показывает похожие заметки на основе косинусной близости. Количественные отчёты подтверждают, что семантический поиск в рамках подготовленных сценариев способен давать более качественное ранжирование, чем базовый keyword search.

## Выводы по главе

В третьей главе была рассмотрена экспериментальная реализация прототипа интеллектуальной системы управления персональными знаниями. Разработанный прототип представляет собой локальное исследовательское Streamlit-приложение, предназначенное для демонстрации и оценки отдельных AI-механизмов, применимых к PKM-системам. Он не является промышленной платформой и не включает функции, выходящие за рамки поставленной исследовательской задачи.

В ходе работы был сформирован синтетический набор данных, включающий 240 русскоязычных PKM-style заметок, распределённых по восьми тематическим группам. Для оценки поисковых методов подготовлены размеченные сценарии, позволяющие автоматически рассчитывать Precision@K, Recall@K, nDCG@K и MRR.

Архитектура прототипа построена на простом разделении модулей загрузки данных, поиска, тегирования, пользовательского интерфейса и экспериментальной оценки. Такая организация позволяет использовать одни и те же поисковые компоненты как в интерфейсе, так и в автоматических экспериментах.

В прототипе реализованы два поисковых подхода: keyword search на основе TF-IDF и semantic search на основе sentence-transformers embeddings. Также реализован режим сравнения, позволяющий отображать результаты двух методов рядом. Дополнительно система поддерживает автоматические теги, автоматическую категорию и подбор похожих заметок с использованием косинусной близости.

Результаты экспериментов показывают, что на подготовленном русскоязычном наборе семантический поиск демонстрирует более высокое среднее значение nDCG при K=3, K=5 и K=10. Это подтверждает применимость embedding-подхода для задач поиска в персональных заметках, особенно при смысловых и переформулированных запросах. При этом выводы следует рассматривать как результаты экспериментальной демонстрации, поскольку используемые данные являются синтетическими.

Таким образом, реализованный прототип подтверждает возможность практического применения методов искусственного интеллекта для поиска, категоризации и связывания заметок в персональной базе знаний. Полученная реализация создаёт основу для дальнейшего исследования интеллектуальных механизмов PKM, не выходя за рамки компактного и воспроизводимого исследовательского приложения.
